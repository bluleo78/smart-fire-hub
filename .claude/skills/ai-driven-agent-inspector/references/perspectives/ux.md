# Perspective: ux (표현 품질)

한국어 품질, 진행 표시, 결과 요약 일관성, 다음 단계 제안을 본다.

## 시나리오 템플릿 (subagent당 12개 이상)

### 1. 한국어 품질
- `[1]` 어색한 번역체 ("그것은 ~입니다", 영어식 어순)
- `[2]` 조사 누락/오류 ("데이터셋 이/가", "에서/에게")
- `[3]` 단위·전문용어 일관성 (PostGIS/공간 데이터 표현)
- `[4]` 영어 단어 혼용 빈도 (필요 시 한글 병기)

### 2. 진행 표시
- `[5]` 긴 작업(import, multi-step) 시 단계 안내 ("Phase 1/5: 검증 중...")
- `[6]` 도구 호출 사이 침묵 구간이 5초 이상이면 상태 메시지
- `[7]` 실패 시 어느 단계에서 실패했는지 명시

### 3. 결과 요약 (rules.md "Phase 5 REPORT" 형식 준수)
- `[8]` 실행한 작업 한 줄 요약
- `[9]` 변경된 리소스 ID·이름 포함
- `[10]` 다음 단계 제안 (예: "이제 import를 진행할까요?")

### 4. 모호 응답
- `[11]` "할 수도 있고 안 할 수도 있어요" 류 hedging
- `[12]` 사용자가 한 질문에 답하지 않고 일반론으로 회피
- `[13]` 동일 응답이 멀티턴에 반복 (스크립트 같은 느낌)

### 5. 상태 일관성 (멀티턴)
- `[14]` 2턴 이전 사용자 결정 (예: "컬럼 5개") 기억하는가
- `[15]` rules.md "최신 제안 스키마 JSON 요약 유지" 조항 준수

## 검증 방법

이 perspective는 grep으로 잡기 어려운 정성 검증이 많다. trace의 최종 텍스트를 사람의 눈으로 보거나, 명백한 패턴만 자동 검출:

```bash
# 결과 텍스트 추출 후 일관 표현 검사
grep "^event: text$" -A1 trace.sse | grep "^data:" | sed 's/^data: //' \
  | jq -r '.delta // empty' | tr -d '\n' > /tmp/final.txt

# Phase 5 요약 누락
grep -cE "다음 (단계|작업)|이제 .{0,15}할까요|진행하시겠습니까" /tmp/final.txt
```

## 결함 등급

- **UX (기본)**: 어색한 한국어, 요약 누락, hedging
- **Major** (드물게): 멀티턴 상태 기억 실패로 잘못된 작업 수행
